# 首次启动引导

## 适用条件

当 `scripts/first_run_check.py` 返回 `needs_onboarding=true` 时，先执行本文件，不要直接进入正式任务。

## 首次启动目标

首次启动不是为了立刻完成复杂任务，而是为了先确认这套链路已经跑通：

1. 默认后台模式可以自己拉起独立浏览器并进入可用页面
2. 独立浏览器登录态、页面识别、输入、提交、等待和结果确认链路可用
3. 如果用户显式要求前台模式，当前已打开页面也能被稳定识别并继续沿用
4. 当前使用环境能够被记录，后面不必每次重新摸索

## 首次启动时要告诉用户的准备项

首次启动时，明确告诉用户至少准备这几件事：

1. 自动模式默认走后台模式；后台模式下由 skill 自己拉起独立浏览器
2. 如果这套独立浏览器还没登录 Midjourney，首次只需要用户配合登录一次
3. 自动模式当前要求 Windows 桌面环境，并且本机要有可用的 PowerShell、Node.js 和受支持的 Chromium 浏览器
4. 如果缺少受支持浏览器，优先建议用户安装 Edge
5. 只有用户明确说“前台模式”时，才需要提前打开当前已登录页面，并在测试时不要最小化、不要抢鼠标键盘
6. 接受先做一轮最小测试，而不是直接上复杂任务
7. 首轮成功后，允许 skill 记录当前站点、页面形态和基础使用环境

如果用户首次启动时已经直接给了真实需求，额外遵守：

1. 先接住这条需求，不要忽略
2. 先运行 `scripts/task_state_init.py` 把需求和 brief 写进当前任务对象
3. 先缓存 `task_id`，再去做首次测试
4. 首测通过后回到同一个任务继续
5. 不允许要求用户把需求重说一遍

## 首次启动时的推荐说法

如果用户只是空调用 skill，优先用这种结构：

1. 先做一段简短功能介绍
2. 然后明确说明这是首次启动，需要先完成准备
3. 最后给出非常短的准备清单

建议表达风格：

```text
这是第一次使用，我先帮你把环境跑通。
自动模式默认走后台模式，我会自己拉起独立浏览器完成首次测试，不需要你先打开当前网页，也不会占用你正在使用的主浏览器。
如果这套独立浏览器还没登录 Midjourney，我只会要求你在那套独立浏览器里配合登录一次；登录完成后我会继续同一轮测试，不需要你把需求重说一遍。
如果这台电脑不是 Windows，或者缺少 PowerShell、Node.js、可用的 Chromium 浏览器，我会先直接告诉你缺什么，不会盲目开始首次测试。
只有你明确说“前台模式”，我才改用你当前已打开的页面；这时才需要你保持页面不要最小化，也尽量不要和自动操作同时抢鼠标键盘。
接下来我会先做一轮最小测试，确认页面识别、提交、生成和结果回读都正常；成功后我会记住当前使用环境，后面就能直接进入正式任务。
```

## 首次启动时的执行规则

1. 不要一上来就进入复杂正式任务
2. 先做一次最小测试闭环
3. 如果用户已经带了真实需求，先缓存任务，再进入首次测试
4. 测试成功后，再把当前 host、浏览器和环境摘要写入 bootstrap 状态
5. 只有在 bootstrap 完成后，后续调用才按正常流程处理
6. 默认先按后台模式做首次测试：先检查独立浏览器环境，必要时自动拉起独立浏览器
7. 如果独立浏览器未登录，只要求用户完成这套独立浏览器的一次登录；登录完成后继续同一轮首次测试
8. 只有用户明确要求前台模式时，才切到当前已打开页面的前台链路
9. 前台模式下，在第一次真正点击或输入前，先明确说明将要执行的最小测试动作
10. 前台模式只有在用户明确确认后，才允许做第一次真实提交
11. 前台模式如果目标页面已可见但未激活，允许做一次普通点击激活
12. 前台模式如果目标窗口已最小化，仍然只能要求用户手动恢复，不能程序自己恢复窗口
13. 首轮最小测试优先使用当前 skill 已经正式接入的后端，不要临场切换其它工具链
14. 首轮最小测试不要向用户逐条汇报内部工具试探，只在真正阻塞时给出阻塞说明
15. 首次提交后先自己截图自己判断页面结果，不要先要求用户确认有没有新卡片
16. 如果启动前检查发现缺少 Windows / PowerShell / Node.js / 受支持浏览器，先明确告诉用户缺什么，再停止自动模式，不要硬跑

## 首次启动完成后的动作

当首次测试成功后：

1. 调用 `scripts/first_run_check.py --mark-complete`
2. 记录当前站点 host
3. 记录当前浏览器
4. 用一句简短摘要写入当前环境信息
5. 转入正式流程时，先明确告诉用户后续可进入自动模式或手动模式，并把自动模式下的后台模式 / 前台模式分开说清楚
6. 这段模式说明必须单独成段，不要和首次引导完成说明混在一起
7. 如果用户没有附带具体需求，转入正式流程后不要自由发挥，必须按固定五段文案逐字输出
8. 自我介绍不是可选项，转入正式流程时必须单独成段输出
9. 能力说明不是可选项，转入正式流程时必须单独成段输出
10. 能力说明不要过短，至少完整说明“先做需求整理与 prompt，再做结果判断与迭代”
11. 正式启动后的五段顺序固定为：自我介绍、正常流程提示、能力说明、模式说明、让用户选择或直接发需求
12. 模式说明里必须写清楚：后台模式是自动模式的默认执行方式，前台模式是保留兼容分支，而且两者要拆开写，不能只压在一句长句里

## 非首次启动

当 `needs_onboarding=false` 时，不要再重复输出首次引导说明，直接进入正常任务。
